# Usage Guide

This document focuses on the recommended high-level API exposed by `easyhash`.

## Default Path

`Hash(password, opts...)` now defaults to `PBKDF2-SHA256`.

```go
hash, err := easyhash.Hash(password)
ok, err := easyhash.Verify(password, hash)
```

The encoded output uses the easyhash self-describing format:

```text
$easyhash$v=1$pbkdf2-sha256$i=100000,l=32$<salt>$<hash>
```

## Select A Different Algorithm

You can switch algorithms explicitly when a project wants a different default:

```go
hash, err := easyhash.Hash(password, easyhash.WithArgon2id())
hash, err := easyhash.Hash(password, easyhash.WithBcryptCost(12))
hash, err := easyhash.Hash(password, easyhash.WithScrypt())
```

## Inspect Stored Hashes

Use `Identify` when you need to branch migration behavior or log diagnostics:

```go
algorithm, err := easyhash.Identify(storedHash)
```

`Verify` accepts both the new easyhash format and the legacy low-level stored formats generated by:

- `CreatePBKDF2`
- `CreateArgon2`
- `CreateScrypt`
- `CreateBcrypt`
- `CreateMD5` for legacy-only compatibility

## Rehash Policy

Use `NeedsRehash` to decide whether an existing stored hash should be replaced:

```go
need, err := easyhash.NeedsRehash(storedHash, easyhash.DefaultPolicy())
```

Typical reasons:

- the stored hash uses a legacy low-level format
- the stored hash uses a different preferred algorithm
- the stored hash uses weaker parameters than the current policy

## Verify And Upgrade

The recommended login migration path is `VerifyAndUpgrade`:

```go
ok, newHash, upgraded, err := easyhash.VerifyAndUpgrade(
    password,
    storedHash,
    easyhash.DefaultPolicy(),
)
```

When `upgraded` is `true`, persist `newHash` back to storage.

## Compatibility Notes

- `DefaultSalt` still exists for backwards compatibility with the historical low-level API behavior.
- New integrations should prefer the high-level API rather than building storage formats directly.
- `MD5` should only be used when verifying old data that cannot be migrated all at once.
